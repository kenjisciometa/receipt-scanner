## 2. 製品・技術

### Scan & Go（モバイルセルフチェックアウト）

**利用フロー:**
1. スマホのブラウザまたはアプリを起動（**アプリDL不要**、Webアプリで利用可）
2. 店内の商品バーコードをスマホカメラでスキャン
3. 商品情報・プロモーションがリアルタイム表示
4. Apple Pay / Google Pay等で決済
5. そのまま退店（レジ不要）

**技術的特徴:**
- Apple App Clips対応（NFCタグ/QRから即起動）
- オフラインモード対応
- チェックアウト時間: 従来比**最大80%短縮**
- 導入期間: **最短2〜4週間**
- 追加ハードウェア不要

### セルフチェックアウトキオスク

- バーコードスキャン型とRFID型の2種類
- RFID版: 商品を置くだけで一括認識（NRF 2023で発表）
- 従来機の**75%サイズ**（省スペース）

### モジュラーPOSプラットフォーム

単一製品からモジュラー戦略へ進化:
- Scan & Go / Self-Checkout Kiosk / mPOS / Cash Desk / RFID Self Checkout
- 全モジュールがシームレスに連携。小売業者は必要に応じて組み合わせ可能

### 不正防止（Trust Engine）

- 特許出願中の機械学習ベースアルゴリズム
- 未決済退店時に店舗アラームが作動
- MishiPay導入後のシュリンク（万引きロス）増加なしと主張

### 決済システム詳細

#### 対応決済手段

| カテゴリ | 対応 |
|---|---|
| デジタルウォレット | Apple Pay、Google Pay、AliPay |
| カード | 主要クレジット/デビットカード（EMV、NFC、磁気ストライプ） |
| BNPL | Klarna、Tabby、Tamara |
| その他 | PayPal、QRコード決済、ギフトカード、ストアクレジット |
| 従来型 | 現金（キオスク/mPOS時）、分割払い対応 |

#### 決済アーキテクチャ — プロセッサー非依存

MishiPay自体は決済処理会社**ではなく**、**決済オーケストレーションレイヤー**として機能する。小売業者が既に使っている決済プロバイダーにそのまま接続する仕組み。

公式統合済みプロセッサー:

| プロセッサー | 備考 |
|---|---|
| **Adyen** | グローバル大手 |
| **Stripe** | 対応済み |
| **Worldpay** | グローバル |
| **PayPal** | デジタル決済 |
| **Verifone** | 決済端末 |
| **Viva Wallet** | 欧州 |
| **Network International** | UAE/中東 |
| **Square** | mPOS |
| **Chase / Ingenico** | F&B |

新規プロセッサーの追加統合もリクエスト可能。セキュリティはPCI DSS Level 1（最高レベル）、SOC 2、ISO 27001、GDPR準拠。

#### ユーザーモードとカード保存

MishiPayには2つの利用モードがあり、カード保存は登録ユーザーのみ:

| モード | カード保存 | 購入履歴 | 摩擦 |
|---|---|---|---|
| **ゲスト利用**（QRスキャン→即利用） | 不可 | なし | 最小（登録不要） |
| **登録ユーザー** | 可能 | あり | アカウント作成が必要 |

#### カード保存の技術的仕組み

- カード情報はMishiPayのサーバーには**一切保存されない**
- カード情報はデバイスから決済プロセッサー（Stripe.js、Adyen Web Drop-in等）に**直接暗号化送信**
- プロセッサー側でトークン化して保管
- MishiPayは**トークン参照・下4桁・有効期限のみ**保持
- 保存カードで支払う場合でも**CVVは毎回再入力が必要**（SCA/セキュリティ要件）

MishiPayが「アプリ不要・登録不要」を最大の売りにしている以上、カード保存はリピーター向けの補助機能であり、メインの訴求ポイントではない。

## 5. MUJIパートナーシップ詳細

### 導入経緯

| 項目 | 詳細 |
|------|------|
| **開始時期** | 2020年秋 |
| **初期導入** | ロンドン・キングスロード店 |
| **全店展開** | 2020年10月下旬に英国全店舗へ |
| **統合期間** | わずか**6週間**で完全統合完了 |

### 技術統合

- iOS/Androidネイティブアプリ + Webアプリの3チャネル
- MUJIバックエンド（商品カタログ・プロモーション）と統合
- リアルタイムプロモーション表示
- Apple App Clips対応（2021年4月〜）

### 成果指標

| 指標 | 結果 |
|------|------|
| **客単価** | 従来チェックアウト比 **+25%** |
| **取引占有率** | 店舗取引の**過半数**がMishiPay経由 |
| **レジ削減** | 2フロア各2台必要だったレジが、1フロアは完全MishiPay運用に |

### キオスク展開

- Carnaby店: MishiPayセルフチェックアウトキオスクを導入（従来機の75%サイズ）
- Islington店: リニューアル時に2台追加

### MUJIにとっての戦略的意義

欧州のMUJI店舗は比較的小規模。レジスペース削減と人件費最適化が重要課題。MishiPayにより売場面積を最大化しつつ、スタッフを接客に再配置。

## 9. 制約事項・課題

### 技術的制約

- **バーコード依存**: 商品にバーコードがあることが前提（量り売い・生鮮は限定的。RFID版で一部解消）
- **スマートフォン必須**: デジタルデバイドの問題
- **インターネット接続**: オフラインモードはあるが基本的に接続環境が望ましい

### ビジネス上の課題

- **売上規模**: 170万〜300万ドルとまだ小規模。収益化は道半ば
- **従業員68名**: 大規模導入時のサポート体制に懸念の可能性
- **競合激化**: Amazon、Toshiba、NCR等大手が参入
- **万引き問題**: セルフチェックアウト利用者の27%が意図的に未スキャン経験あり（LendingTree調査、2023年の15%から急増）

## 10. SciPOSとの比較・営業での活用ポイント

### MishiPayとSciPOSの位置づけの違い

| 項目 | MishiPay | SciPOS |
|------|----------|--------|
| **対象業態** | 物販（リテール） | 飲食（レストラン） |
| **主機能** | セルフチェックアウト（決済） | 注文→調理→提供→決済の全フロー |
| **テーブルマップ** | なし（物販なので不要） | あり（座席管理・配膳効率化） |
| **キッチン連携** | なし | あり（KDS） |
| **BYOD** | 顧客のスマホ | 店舗のタブレット |
| **共通点** | セルフサービスでスタッフを接客に再配置、客単価向上 | 同左 |

### 営業での活用

- MishiPayは**競合ではなく参照事例**。物販で証明された「セルフサービス→スタッフ再配置→客単価向上」の論理を、飲食に応用するストーリーに使う
- MishiPayの成功を褒めることで、MUJI側の心理的ハードルを下げる
- **「MishiPayが物販でやっていることを、飲食ではテーブルマップ付きで実現する」** という説明が最もシンプル
