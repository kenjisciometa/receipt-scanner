## 2. 顧客チャネル: Wallet Pass ファースト

### 2.1 なぜWallet Passか

業界調査（14社比較、2025-2026年トレンド分析）の結果、**PWAではなくApple/Google Wallet Passが最適解**と判断。

| 指標 | Wallet Pass | PWA | ネイティブアプリ |
|------|-----------|-----|--------------|
| **登録率** | **65-75%** | 30-50% | 10-20% |
| **Push通知開封率** | **99%** | 5-20% | 30-40% |
| **オフライン表示** | **対応** | 限定的 | 対応 |
| **iOS制限** | **なし** | Web Push制限あり | なし |
| **インストール** | 1タップで追加 | ブックマーク的 | App Store経由 |
| **開発工数** | 小〜中 | 中 | 大 |

**業界実績**:
- Portillo's（Wallet Pass Only）: 10ヶ月で200万人登録、売上の10%がロイヤルティ経由
- PAR Technology: 「Smart Passes」をWallet Pass専用で発表
- 5Loyalty, Leat, Fivestars/SumUp: 全てWallet Pass採用

### 2.2 競合14社の顧客チャネル分布

| 方式 | 企業 | 割合 |
|------|------|------|
| **Wallet Pass** | 5Loyalty, Leat, Fivestars/SumUp, Stamp Me | 29% |
| **QR → ブラウザ** | Honei, sunday, CloudWaitress, dinii | 29% |
| **電話番号/SMSのみ** | Square, Toast, Loyverse, Popmenu | 29% |
| **ネイティブアプリ** | Flipdish, Craver, Stamp Me | 21% |

※複数方式を併用する企業あり。ネイティブアプリ専用は大手チェーン向けのみ。

### 2.3 SciPOSの採用方式

```
プライマリ:  Apple Wallet / Google Wallet Pass
フォールバック: QR → ブラウザ（Wallet非対応端末用）
```

**全案（A/B/C）共通でこの方式を採用する。**

## 4. 案A: ミニマム（デジタルスタンプカードのみ）

### 4A-1. 概要

紙のスタンプカードのデジタル版。顧客はアプリDL不要、QRスキャンまたはNFCタップでWalletにカード追加。来店ごとにスタンプ取得、規定回数で特典。

### 4A-2. 機能一覧

| 機能 | 説明 |
|------|------|
| **スタンプカード作成** | レストランがPOS管理画面で設定（例: 10回来店で1食無料） |
| **Wallet Pass発行** | 初回スキャン時にApple/Google Walletにカードを自動追加 |
| **スタンプ付与** | 会計時にQRコード表示 or NFCタップ → Wallet Pass上のスタンプ更新 |
| **スタンプ確認** | 顧客はWalletアプリでスタンプ数を常時確認（オフライン可） |
| **Wallet Push通知** | スタンプ付与時・特典到達時にWallet経由で通知（開封率99%） |
| **特典引き換え** | 規定スタンプ数到達時にPOS画面に通知 → スタッフが特典適用 |
| **複数カード対応** | 1店舗で複数のスタンプカード運用可能（ランチ用/ディナー用等） |
| **有効期限** | カードごとに有効期限設定可能（Wallet上で残日数表示） |
| **QRフォールバック** | Wallet非対応端末はQR → ブラウザ画面でスタンプ確認 |

### 4A-3. 顧客フロー

```
【Wallet Pass対応端末（大多数）】

来店 → 会計時にレシートのQR or テーブルNFCをスキャン
  ↓
初回: 「Walletに追加」ボタン → 1タップでApple/Google Walletに保存
  ↓
スタンプ+1 → Walletカードが自動更新（Push通知: 「3/10スタンプ達成！」）
  ↓
10回目: Wallet通知「特典利用可能！」→ 次回来店時にWalletカード提示 → POSで適用


【Wallet非対応端末（フォールバック）】

来店 → QRスキャン → ブラウザで電話番号入力
  ↓
スタンプ+1 → ブラウザ画面に残りスタンプ数表示
  ↓
次回: 同じQRスキャン or 電話番号でPOS側から検索
```

### 4A-4. 技術設計

| 要素 | 方針 |
|------|------|
| 顧客識別 | 電話番号ベース（Wallet Pass内部にも紐付け） |
| Wallet Pass生成 | Apple: PassKit (.pkpass) / Google: Google Wallet API |
| Pass更新 | サーバーサイドでPass更新 → APNs/Google Push経由で端末に自動反映 |
| データ保存 | Supabase（既存インフラ活用） |
| スタンプ付与トリガー | POS会計完了時に自動QR生成 / NFC書き込み |
| フォールバック | Wallet非対応時はブラウザ画面（レスポンシブWeb） |
| 不正防止 | 1日1スタンプ上限 / 最低注文金額条件 / 会計紐付け必須 |
| Pass生成ツール | PassKit SDK / 自前実装（Apple PassKitは仕様公開済み） |

### 4A-5. Wallet Pass の仕様

| 項目 | Apple Wallet | Google Wallet |
|------|-------------|---------------|
| ファイル形式 | .pkpass (ZIP) | Google Wallet Object (API) |
| カスタマイズ | ロゴ、背景色、バーコード、テキストフィールド | 同左 + Hero画像 |
| Push更新 | APNs経由でリアルタイム更新 | Google API経由 |
| バーコード | QR / PDF417 / Aztec | QR / PDF417 |
| ロック画面表示 | 位置情報ベースで自動表示（店舗近くで） | 同左 |
| オフライン | 完全対応（最終同期状態を表示） | 完全対応 |
| 開発コスト | 無料（Apple Developer Program内） | 無料（Google Wallet API） |

### 4A-6. メリット・デメリット

| メリット | デメリット |
|---------|-----------|
| 開発工数最小（2-3ヶ月） | 顧客データの深さが浅い（名前+連絡先のみ） |
| Wallet Push通知の開封率99% | マーケティング配信機能なし（Wallet通知のみ） |
| アプリDL不要（1タップ追加、登録率65-75%） | 将来的にフル改修が必要になる可能性 |
| オフラインでカード表示可能 | セグメント配信・分析不可 |
| ロック画面に店舗近くで自動表示（再来店促進） | QRフォールバックのUXはやや劣る |
| NFC活用で既存差別化と連動 | Wallet Pass更新のサーバー実装が必要 |
| レストランの運用負荷ゼロ | — |

## 5. 案B: バランス型（スタンプ + ポイント + 顧客DB）

### 5B-1. 概要

案Aの全機能に加え、ポイント制度と基本的な顧客プロファイルを追加。Wallet Passにポイント残高も表示。レストランは顧客の来店履歴・注文傾向を把握でき、Wallet Push通知でセグメント配信可能。

### 5B-2. 機能一覧

| カテゴリ | 機能 | 説明 |
|---------|------|------|
| **スタンプ** | 案Aの全機能 | デジタルスタンプカード（Wallet Pass） |
| **ポイント** | ポイント付与 | 注文金額に応じてポイント自動付与（例: €1 = 1pt） |
| **ポイント** | ポイント交換 | 貯まったポイントを割引/特典に交換 |
| **ポイント** | Wallet表示 | Wallet Pass上にポイント残高・ティアをリアルタイム表示 |
| **ポイント** | ティア（ランク） | ブロンズ/シルバー/ゴールド等の段階制（オプション） |
| **顧客DB** | プロファイル | 名前、連絡先、来店回数、累計利用額、最終来店日 |
| **顧客DB** | 注文履歴 | 過去の注文内容を紐付け保存 |
| **顧客DB** | タグ/メモ | スタッフが顧客にタグ付け（VIP、アレルギー等） |
| **通知** | Wallet Push | スタンプ/ポイント更新、特典到達、ランクアップ時の即時通知 |
| **通知** | セグメント配信 | 基本的なセグメント別Wallet Push（例: 30日未来店者に再来店促進） |
| **通知** | 自動トリガー | 誕生日/ランクアップ時の自動通知 |
| **分析** | ダッシュボード | リピート率、顧客LTV、来店頻度分布 |

### 5B-3. 顧客フロー

```
【初回来店】
会計完了 → QR/NFC → 「Walletに追加」1タップ
  ↓
Wallet Pass追加完了（スタンプ1/10 + ポイント残高表示）
  ↓
Wallet通知: 「あと3回でゴールド会員！」

【リピート来店】
会計完了 → Walletカード提示 or QR/NFC → 自動識別
  ↓
POS画面: 「ゴールド会員 / 来店12回目 / 前回注文: マルゲリータ」
  ↓
スタッフ: 「○○さん、いつもありがとうございます」
  ↓
Wallet自動更新: スタンプ+1 / ポイント+N

【30日未来店】
Wallet Push通知: 「お久しぶりです！次回来店でポイント2倍」
（開封率99% — メールの20倍）

【店舗付近を通過】
ロック画面にWalletカードが自動表示（位置情報トリガー）
→ 「今日のランチはいかがですか？ポイント2倍実施中」
```

### 5B-4. 技術設計

| 要素 | 方針 |
|------|------|
| 顧客識別 | 電話番号 + Wallet Pass ID（デバイス紐付け） |
| データ保存 | Supabase: customers / loyalty_cards / points / visits テーブル |
| ポイント計算 | POS会計完了Webhook → ポイント自動付与 → Wallet Pass更新Push |
| Wallet更新 | スタンプ/ポイント変更時にAPNs/Google API経由でPass自動更新 |
| セグメント配信 | Wallet Push通知でセグメント別メッセージ送信 |
| 位置情報トリガー | Wallet Pass内にLocation設定 → 店舗付近でロック画面表示 |
| POS統合 | 会計画面に顧客情報バナー表示 |
| フォールバック | Wallet非対応: ブラウザ画面 + SMS通知 |

### 5B-5. データモデル（概要）

```
customers
├── id (UUID)
├── phone (UNIQUE, 識別キー)
├── email
├── name
├── wallet_pass_id (Apple/Google PassのシリアルNo.)
├── wallet_device_token (Push通知用)
├── created_at
├── tags[] (VIP, アレルギー等)
└── org_id (FK → organizations)

loyalty_cards
├── id
├── customer_id (FK)
├── card_type (stamp / points)
├── current_stamps / current_points
├── tier (bronze / silver / gold)
├── wallet_pass_serial (PassKit連携用)
└── expires_at

visits
├── id
├── customer_id (FK)
├── order_id (FK → orders)
├── visited_at
├── amount_spent
└── points_earned

rewards
├── id
├── org_id
├── name ("ドリンク1杯無料")
├── cost_points / cost_stamps
├── type (discount_percent / free_item / custom)
└── active (boolean)
```

### 5B-6. メリット・デメリット

| メリット | デメリット |
|---------|-----------|
| スタンプ+ポイント+顧客DBで競合の中堅レベルに到達 | 開発4-5ヶ月（案Aの約2倍） |
| Wallet Push通知（開封率99%）でセグメント配信 | Wallet Pass更新APIの実装が案Aより複雑 |
| 位置情報トリガーで店舗付近の顧客に自動リーチ | レストラン側に最低限の運用意識が必要 |
| 顧客データ蓄積 → 将来のAI基盤 | ティア制はルール設計が複雑 |
| POS画面での顧客認識はスタッフ体験を向上 | — |
| 段階的に案Cへ拡張可能な設計 | — |

## 6. 案C: フルCRM + マーケティング自動化

### 6C-1. 概要

案Bの全機能に加え、本格的なCRM機能とマーケティング自動化を搭載。diniiのLINE CRM（0.5%チャーン）やSevenRooms（100+統合）に匹敵する顧客管理を目指す。Wallet Push通知に加え、メール/SMSチャネルも追加。

### 6C-2. 追加機能（案Bの上に）

| カテゴリ | 機能 | 説明 |
|---------|------|------|
| **CRM深化** | アレルギー/食事制限 | 顧客の食事制限を注文時に自動フラグ表示 |
| **CRM深化** | 好み学習 | 注文履歴から自動で好みを推定（辛い料理好き等） |
| **CRM深化** | 記念日管理 | 誕生日/結婚記念日の登録と自動リマインド |
| **CRM深化** | 家族/グループ | 複数顧客をグループ化（家族来店パターン把握） |
| **マーケティング** | メール自動配信 | テンプレートベースのメールキャンペーン |
| **マーケティング** | SMS配信 | セグメント別SMS（Twilio/MessageBird連携） |
| **マーケティング** | マルチチャネル配信 | Wallet Push + メール + SMSの最適チャネル自動選択 |
| **マーケティング** | 自動シナリオ | 条件トリガーの自動配信（来店後24h感謝/30日未来店/誕生日前7日） |
| **マーケティング** | クーポン生成 | 使い捨てクーポンコードの自動生成 → Wallet Passに直接配布 |
| **マーケティング** | A/Bテスト | メッセージの効果比較テスト |
| **分析** | RFMスコアリング | 顧客を自動でセグメント（優良/離脱危機/休眠等） |
| **分析** | LTV予測 | 顧客別の生涯価値予測 |
| **分析** | キャンペーンROI | 配信施策ごとの売上貢献度分析 |
| **連携** | Google口コミ促進 | 満足度の高い顧客に自動でレビュー依頼 |
| **連携** | SNS連携 | Instagram/Facebook投稿でポイント付与 |

### 6C-4. メリット・デメリット

## 7. 推奨: 段階的実装アプローチ

### Phase分割で案B構成を段階リリース

```
Phase 1 (2-3ヶ月) — 案Aリリース
├── デジタルスタンプカード
├── QR/NFC → Wallet Pass自動追加
├── Apple Wallet / Google Wallet 対応
├── Wallet Push通知（スタンプ更新/特典到達）
├── 基本的な顧客登録（電話番号+名前）
├── QRフォールバック（Wallet非対応端末用ブラウザ画面）
└── ★ 最小工数で「ロイヤルティ機能あり」を訴求
     ★ 登録率65-75%、Push開封率99%を初日から実現

Phase 2 (+2ヶ月) — 案Bへ拡張
├── ポイント制度追加（Wallet Pass上にポイント残高表示）
├── 顧客DB/プロファイル
├── ティア/ランク制度（Wallet Passデザインがランクで変化）
├── POS画面での顧客情報表示
├── 位置情報トリガー（店舗付近でロック画面表示）
├── セグメント別Wallet Push通知
└── ★ 競合の中堅レベルに到達

Phase 3 (+3ヶ月) — 案Cの一部を選択導入
├── 自動シナリオ（離脱防止/誕生日/近接リマーケティング）
├── メール/SMS配信（マルチチャネル）
├── RFMセグメント
├── クーポンのWallet Pass内配布
├── Google口コミ促進
└── ★ PREMIUM価格の付加価値を強化
```

### なぜこのアプローチか

1. **Phase 1で即座に「CRM機能搭載」を市場に訴求できる**（競合との比較表で✅が付く）
2. **Wallet Passの登録率65-75%で初日から大量の顧客データ蓄積が開始**
3. **Push開封率99%により、Phase 2/3の通知機能の効果が桁違いに高い**
4. **市場フィードバックを得てから機能を深化**（過剰開発を防止）
5. **Phase 1のデータモデルをPhase 2/3で再利用**（設計段階で先を見据える）
