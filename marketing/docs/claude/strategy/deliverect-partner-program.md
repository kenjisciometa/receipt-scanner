## 5. 「Locations Served」フィルタリング問題

### 判明している事実

開発者フォーラムでの投稿:

> *「deliverect.com/en/become-a-partner のパートナー申請フォームで 'Locations Served' の項目でフィルタリングされ、弾かれた。我々はプレローンチ段階だ」*
> *「100+拠点になるまで待つしかないのか？」*

### 分析

| 項目 | 状況 |
|------|------|
| 公式ドキュメントに最低拠点数の記載 | **なし** |
| フォームに「Locations Served」入力欄 | **あり** |
| 少数拠点で弾かれた事例 | **あり**（プレローンチ企業の報告） |
| 「100拠点」が公式基準か | **不明**（ユーザー推測の可能性） |
| 公式のパートナー要件の公開 | **なし**（ケースバイケースと推測） |

### 「Locations Served」の意味

**SciometaPOSが導入されているレストランの店舗数**を指す。SciPOS自体の拠点数ではない。

## 8. SciometaPOSにとっての評価

### メリット

| メリット | 説明 |
|---------|------|
| **Woltを迂回できる可能性** | DeliverectはWolt連携を構築済み。SciPOS→Deliverect→Woltの間接連携が可能（Woltの直接承認不要） |
| **マルチプラットフォーム対応** | 1つのAPI連携で Wolt + Uber Eats + Bolt Food + 将来の新プラットフォームに対応 |
| **北欧が対象地域** | パートナーフォームの地域リダイレクトに「Nordics」が明示的に含まれる |
| **標準化されたデータフォーマット** | 各プラットフォーム固有のAPI差異をDeliverectが吸収 |
| **認定テスト環境** | Postman + Dummy POSでの開発・テスト環境あり |

### デメリット・リスク

| デメリット | 説明 |
|-----------|------|
| **Locations Servedフィルタ** | 利用店舗数が少ないとパートナー申請で弾かれる可能性 |
| **レストランの追加コスト** | $119〜249/月がレストランに発生（SciPOS月額に上乗せ） |
| **パートナー費用が不透明** | SciPOS側のAPI利用料・認定費用が非公開 |
| **Wolt間接連携の制限** | 「Woltマーケットプレイス注文→POS受信」方向のみ。「自社注文→Wolt配達」方向（Wolt Drive）はDeliverect経由でもWoltの承認が必要な可能性 |

### 重要な注意点

**Deliverect経由のWolt連携は「Woltマーケットプレイス上の注文をPOSで受信する」方向がメイン**。これは:
- ✅ Woltアプリで注文 → Deliverect → SciPOS KDSに表示 → Woltが配達
- ❓ SciPOS自社注文ページで注文 → Deliverect → Wolt Driveで配達 ← これが可能かは未確認

後者（Wolt Drive経由の配達手配）については、DeliverectがWolt Drive連携を持っていても、**Wolt側がSciPOS経由の配達を受け入れるかは別問題**。
