# スペック文書作成

ユーザーの要件に基づき、実装可能な精度の定義書（スペック）を作成してください。

## 入力

ユーザーから以下のいずれかが提供される:
- 自然言語での機能要件の説明
- 参考となる既存コードやドキュメントの指定
- Google Driveやその他の外部ドキュメントへの参照

## 手順

### 1. 要件の整理とヒアリング

1. ユーザーの要件を把握する
2. 不明点や曖昧な点があれば **最初に質問する**（実装方針、スコープ、優先度）
3. 要件が明確になるまで実装詳細には進まない

### 2. 既存コードベースの調査

1. 関連する既存コード・コンポーネント・APIを調査する
2. プロジェクトの既存パターン・規約を把握する（命名、ディレクトリ構成、状態管理）
3. 再利用できる既存の実装がないか確認する
4. DBスキーマの現状を確認する（Supabaseの場合は `list_tables` を活用）

### 3. スペック文書の作成

以下のテンプレートに沿って `docs/claude/` 配下にMarkdownファイルを作成する。
ファイル名は `{feature-name}-spec.md` の形式とする。

---

## スペックテンプレート

```markdown
# {機能名} 実装定義書

## 概要
- 目的: この機能が解決する課題
- スコープ: 今回実装する範囲（明示的に含むもの・含まないもの）

## ユーザーストーリー
- どのユーザーが
- 何をしたい
- なぜ（ビジネス価値）

## 機能要件

### {機能1}
- 詳細な振る舞いの説明
- 入力 → 処理 → 出力
- エッジケース

### {機能2}
...

## データモデル

### テーブル/スキーマ変更
（新規テーブル、カラム追加、RLSポリシーなど）

```sql
-- マイグレーション例
```

### 型定義
（TypeScript/Dart の型・インターフェース）

## API設計

### エンドポイント（該当する場合）
- メソッド、パス、リクエスト/レスポンス形式

### Supabase RPC/クエリ（該当する場合）
- 使用するクエリパターン

## UI/UX設計

### 画面構成
- 画面一覧とナビゲーションフロー
- 主要コンポーネントの構造

### 状態管理
- 使用するstate/provider/store
- データフローの概要

## 実装計画

### ファイル構成
（作成・変更するファイルの一覧と役割）

### 実装順序
1. ステップ1（依存関係のないものから）
2. ステップ2
3. ...

### 影響範囲
- 既存機能への影響と対応方法

## 非機能要件
- パフォーマンス考慮事項
- セキュリティ考慮事項（RLS、認証）
- エラーハンドリング方針

## テスト方針
- テスト対象と確認方法
- エッジケースのテストシナリオ

## 未決事項
- 追加のヒアリングが必要な項目（あれば）
```

---

### 4. レビューと確認

1. 作成したスペック文書をユーザーに提示する
2. フィードバックを受けて修正する
3. 確定後、ファイルとして保存する

## 注意事項

- **過不足なく書く**: 実装に必要な情報は漏らさず、不要な情報は含めない
- **具体的に書く**: 「適切に処理する」ではなく、具体的な処理内容を記述する
- **既存パターンを尊重する**: プロジェクトの規約から逸脱しない設計にする
- **実装可能な粒度**: 曖昧さを残さず、このスペックだけで実装できるレベルにする
- **スコープを明確に**: 今回やること・やらないことを明示し、スコープクリープを防ぐ

## ワークフロー上の位置づけ

このスペックは以下のスキルと連携する:
1. `/spec` → 定義書作成（このスキル）
2. 実装（手動またはPlan Modeで実施）
3. `/verify-fix` → 定義書との整合性検証・バグチェック
4. `/refactor-check` → コード品質の確認・改善提案
5. `/commit-push` → コミット・プッシュ
