# 実装検証（バグ修正・機能実装の品質チェック）

直近の変更が定義書（スペック）通りに正しく実装されているか、バグや副作用がないかを検証してください。

## 手順

### 1. 変更差分の把握
1. `git diff` と `git diff --cached` で現在の変更を確認する
2. mainブランチとの差分がある場合は `git diff main...HEAD` も確認する
3. 変更されたファイルをすべてリストアップする

### 2. 定義書との整合性チェック
1. 変更に対応する定義書（spec）が存在するか確認する
   - まず `docs/claude/*-spec.md` をglob検索する
   - 見つからない場合は `docs/claude/` 配下を広く検索する
   - 変更内容のキーワード（機能名、テーブル名等）でgrepも行う
2. 定義書が存在する場合、以下を照合する:
   - 定義書に記載された **全要件** が実装されているか（漏れチェック）
   - 定義書に **記載のない機能** が勝手に追加されていないか（スコープ逸脱チェック）
   - データモデル・API・UIの仕様が定義書と一致しているか
   - エッジケースやエラーハンドリングが定義書の指示通りか
3. 定義書が存在しない場合はこのステップをスキップし、その旨を報告する

### 3. バグ検出
変更されたコードを精読し、以下を確認する:
- **ロジックエラー**: 条件分岐の誤り、off-by-one、無限ループ
- **null/undefined安全性**: null参照、optional chainingの欠落
- **型の不整合**: TypeScript型エラー、Dart型エラー、暗黙のany
- **非同期処理**: awaitの欠落、競合状態、エラー未捕捉のPromise/Future
- **状態管理**: 状態の不整合、不要な再レンダリング、リスナーの解除漏れ

### 4. 既存機能の毀損チェック（リグレッション）
1. 変更されたファイルの **変更前の機能** が引き続き動作するか確認する
2. 変更されたファイルを参照している他のファイルを `grep` で特定し、影響範囲を確認する
3. 以下に注意する:
   - 関数シグネチャの変更による呼び出し元の破壊
   - export/importの変更による依存先の破壊
   - DBスキーマ変更に対するクエリの未対応
   - 共有コンポーネントの変更による他画面への影響

### 5. セキュリティ・パフォーマンス
- RLSポリシーの欠落・不備（Supabaseテーブル変更時）
- SQLインジェクション、XSS、認証バイパスの可能性
- N+1クエリ、不要なループ、大量データの非効率な処理

### 6. テスト実行
1. 該当サブプロジェクトのテストを実行する
2. テストが失敗した場合、変更が原因かどうかを分析する

## 報告フォーマット

検証結果を以下の形式で報告する:

```
## 検証結果サマリー

### 定義書との整合性: ✅ OK / ⚠️ 差異あり / ➖ 定義書なし
（差異がある場合は具体的に列挙）

### バグ: ✅ 検出なし / 🐛 検出あり
（検出された場合はファイルパス:行番号付きで報告）

### リグレッション: ✅ 問題なし / ⚠️ リスクあり
（リスクがある場合は影響範囲と内容を報告）

### セキュリティ/パフォーマンス: ✅ 問題なし / ⚠️ 指摘あり

### テスト: ✅ 全パス / ❌ 失敗あり

### 総合判定: ✅ LGTM / 🔧 要修正
```

問題が見つかった場合は、修正案も併せて提示する。
