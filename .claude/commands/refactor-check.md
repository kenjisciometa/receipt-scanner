# リファクタリングチェック

直近の変更差分を対象に、リファクタリングの余地を分析して具体的な改善案を提案してください。

## 手順

### 1. 対象コードの把握
1. `git diff` と `git diff --cached` で現在の変更を確認する
2. mainブランチとの差分がある場合は `git diff main...HEAD` も確認する
3. 変更されたファイルの全体を読み、周辺コードも把握する

### 2. 重複コードの検出（DRY原則）
- 変更ファイル内での **コード重複** を検出する
- 変更ファイルと **他ファイル間** の重複も検索する（grep/globで類似パターンを探索）
- 共通ロジックをユーティリティ関数・共有コンポーネント・mixinに抽出できないか検討する
- ただし、1〜2箇所の類似程度なら抽出しないほうがシンプルな場合もある。過度な抽象化を避ける

### 3. 関数・コンポーネントの責務分離（単一責任原則）
- 1つの関数/メソッドが **複数の責務** を持っていないか
- 関数の行数が長すぎないか（目安: 50行超は分割を検討）
- コンポーネントが肥大化していないか（表示ロジック・ビジネスロジック・データ取得が混在）
- 適切な粒度でのファイル分割ができるか

### 4. 命名と可読性
- 変数名・関数名・クラス名が **意図を正しく伝えている** か
- プロジェクト内の **命名規則と一貫性** があるか（camelCase/snake_case/PascalCase）
- マジックナンバーや文字列リテラルが定数化されているか
- 条件式が複雑すぎないか（早期リターンや変数抽出で簡潔にできるか）
- ネストが深すぎないか（目安: 3段超は改善を検討）

### 5. 型安全性と堅牢性
- **TypeScript**: any型の使用、型アサーション（as）の多用、union型で絞れる箇所
- **Dart**: dynamic型の使用、型推論に頼りすぎている箇所
- null/undefinedの扱いが適切か（optional chaining、nullish coalescing の活用）
- エラーハンドリングが適切な粒度か（catchで握りつぶしていないか）

### 6. パフォーマンスの改善余地
- 不要な再レンダリング・再計算（useMemo/useCallback/const の活用）
- N+1クエリ、ループ内のDB/APIアクセス
- 大量データの不要なコピーやフィルタリング
- 非同期処理の並列化可能な箇所（Promise.all / Future.wait）

### 7. プロジェクトパターンとの一貫性
- プロジェクト内の既存パターン・規約に従っているか
- 既存のユーティリティやヘルパーが活用されているか（車輪の再発明をしていないか）
- import/exportの整理（未使用import、循環参照）
- ディレクトリ構成の規約に沿っているか

## 報告フォーマット

```
## リファクタリング提案

### 優先度: 高 🔴
（バグリスクやメンテナンス性に大きく影響するもの）
- [ファイルパス:行番号] 内容と具体的な改善案

### 優先度: 中 🟡
（可読性・一貫性を改善するもの）
- [ファイルパス:行番号] 内容と具体的な改善案

### 優先度: 低 🟢
（あると良い程度の改善）
- [ファイルパス:行番号] 内容と具体的な改善案

### 提案なし ✅
リファクタリングの必要性は低く、現状のコードで問題ありません。
```

## 注意事項

- 動作しているコードを壊さない提案をする。リファクタリングは **振る舞いを変えずに** 構造を改善すること
- 過度な抽象化・早すぎる最適化は避ける。現時点で明確なメリットがある提案のみ行う
- 提案には **Before/After のコード例** を含め、改善後の姿を具体的に示す
- 変更が不要な場合は「提案なし」と明記する。無理に提案を作らない
